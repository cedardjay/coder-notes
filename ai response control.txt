@ControllerAdvice
public class GlobalExceptionHandler {

    @ExceptionHandler(OurException.class)
    public ResponseEntity<ErrorResponse> handleOurException(OurException e) {
        return ResponseEntity.status(e.getStatus()).body(
            ErrorResponse.builder()
                .status(e.getStatus())
                .message(e.getMessage())
                .timestamp(LocalDateTime.now())
                .build()
        );
    }

    @ExceptionHandler(MethodArgumentNotValidException.class)
    public ResponseEntity<ErrorResponse> handleValidationException(MethodArgumentNotValidException e) {
        String message = e.getBindingResult()
                .getFieldErrors()
                .stream()
                .map(error -> error.getField() + ": " + error.getDefaultMessage())
                .collect(Collectors.joining(", "));

        return ResponseEntity.status(400).body(
            ErrorResponse.builder()
                .status(400)
                .message(message)
                .timestamp(LocalDateTime.now())
                .build()
        );
    }

    @ExceptionHandler(AccessDeniedException.class)
    public ResponseEntity<ErrorResponse> handleAccessDeniedException(AccessDeniedException e) {
        return ResponseEntity.status(403).body(
            ErrorResponse.builder()
                .status(403)
                .message("You are not authorized to perform this action")
                .timestamp(LocalDateTime.now())
                .build()
        );
    }

    @ExceptionHandler(Exception.class)
    public ResponseEntity<ErrorResponse> handleGeneralException(Exception e) {
        return ResponseEntity.status(500).body(
            ErrorResponse.builder()
                .status(500)
                .message("An unexpected error occurred: " + e.getMessage())
                .timestamp(LocalDateTime.now())
                .build()
        );
    }
}

The critical decision point here is what the OurException handler is doing now versus before. Previously it was hardcoded to 404. Now it reads e.getStatus() — which means the exception itself carries the correct status, and the handler just forwards it. This is why updating OurException to carry a status field was the prerequisite for this to work cleanly.

Three decisions worth evaluating beyond the mechanical refactor.

grantRole and the IllegalArgumentException tradeoff. This is the one method that can't simply propagate freely — Role.valueOf() throws a standard Java IllegalArgumentException which your GlobalExceptionHandler would catch as a generic 500. That's wrong; an invalid role name is a client error, not a server error. The solution is to catch it locally and rethrow as OurException with 400. This is the pattern for when you need to translate an exception from one semantic category to another, not suppress it.

register status was 200, now should be 201. The original code had response.setStatusCode(200) which was already semantically wrong — your controller refactor already corrected this to ResponseEntity.status(201). The service no longer owns that concern, so the mistake is gone naturally.

grantRole and deleteUser now return void. The original returned Response<Void> and set success messages like "Role updated successfully" — which the conversation history already established as noise the frontend ignores. The methods now signal success through the absence of an exception, which is what void means in the clean pattern: if it returns, it worked.

@Transactional
public void investInLoan(Long loanRequestId, InvestRequest investmentRequest, String email) {
    // --- FETCH ---
    User investor = userRepository.findByEmail(email)
            .orElseThrow(() -> new OurException("User not found", 404));

    LoanRequest loanRequest = loanRequestRepository.findById(loanRequestId)
            .orElseThrow(() -> new OurException("Loan request not found", 404));

    // --- VALIDATE ---
    if (loanRequest.getBorrower().getId().equals(investor.getId())) {
        throw new OurException("You cannot invest in your own loan request", 400);
    }

    if (loanRequest.getStatus() != LoanStatus.APPROVED &&
            loanRequest.getStatus() != LoanStatus.PARTIALLY_FUNDED) {
        throw new OurException("Loan request is not open for investment", 400);
    }

    BigDecimal remaining = loanRequest.getRequestedAmount()
            .subtract(loanRequest.getAmountFunded());

    if (investmentRequest.getAmount().compareTo(remaining) > 0) {
        throw new OurException("Investment amount exceeds remaining capacity of " + remaining, 422);
    }

    // --- EXECUTE ---
    BigDecimal newAmountFunded = loanRequest.getAmountFunded()
            .add(investmentRequest.getAmount());

    LoanStatus newStatus = newAmountFunded.compareTo(loanRequest.getRequestedAmount()) == 0
            ? LoanStatus.FULLY_FUNDED
            : LoanStatus.PARTIALLY_FUNDED;

    // --- PERSIST ---
    MatchedRequest match = new MatchedRequest();
    match.setLoanRequest(loanRequest);
    match.setInvestor(investor);
    match.setInvestorAmount(investmentRequest.getAmount());
    match.setMatchDate(LocalDateTime.now());
    matchedRequestRepository.save(match);

    loanRequest.setAmountFunded(newAmountFunded);
    loanRequest.setStatus(newStatus);
    loanRequestRepository.save(loanRequest);
}

Yes — and there are several layers to this, each solving a different documentation problem. Let me break them down analytically.
The layers of application documentation

API Specification — OpenAPI/Swagger

This is the closest thing to a source of truth for what your API contracts are. It documents every endpoint, request shape, response shape, status codes, and authentication requirements. In Spring Boot you add a dependency and annotate your controllers — Swagger UI generates an interactive interface automatically.

The critical distinction: OpenAPI documents what the API promises, not how it delivers it. It's the contract layer.

Architecture Decision Records (ADRs)

These document why decisions were made, not just what was decided. A typical ADR answers: what was the problem, what options were considered, what was chosen, and what tradeoffs were accepted.

Your conversation history is actually a crude form of ADRs — "why did we remove Response<T>", "why does investInLoan return 201 not 204". Professional teams write these formally so future developers understand the reasoning, not just the outcome.

The evaluation tradeoff: code shows what the system does. ADRs show why it does it that way. Without ADRs, the next developer has no way to distinguish a deliberate decision from an accident.

Entity-Relationship Diagrams and Domain Models

These document the data layer — how entities relate, what constraints exist, what the business domain looks like structurally. Your LoanRequest, MatchedRequest, User, Transaction, RepaymentSchedule relationships would live here.

Runbooks and Operational Documentation

These document how the system behaves in production — how to deploy, what monitoring exists, how to respond to specific failures. Less relevant at your stage but critical before anything goes live.
The source of truth problem

The deeper evaluation worth making: no single document is truly the source of truth, because each layer can drift from the others.

The code is the only thing that actually runs — so in one sense, the code is the source of truth. But code without context doesn't explain intent, constraints, or rejected alternatives.

Code          → what actually happens
Tests         → what was intended to happen  
OpenAPI spec  → what the contract promises
ADRs          → why the contract was designed that way

When these conflict, the code wins — but the conflict itself signals documentation debt.

That's two distinct financial operations that your domain model needs to differentiate clearly.

Fund disbursal — money flows from the platform to the borrower. One event, one transaction, triggered manually by admin when status is FULLY_FUNDED.

Loan repayment collection — money flows from the borrower to the platform. This happens on a schedule — once per repayment period, triggered either manually or automatically. This is what your RepaymentSchedule entries represent: each row is a payment due.

Investor payout — money flows from the platform to the investor. This is the third operation you haven't modeled yet. It can be structured two ways and the evaluation matters:

Option A — Payout per repayment. Every time a borrower makes a repayment, the platform immediately distributes the investor's share. If there are multiple investors with different contribution percentages, each gets their proportional cut immediately.

Option B — Payout at loan completion. The platform holds collected repayments and distributes to investors only when the loan is fully repaid.

The evaluation tradeoff is business model dependent. Option A is more complex to implement — you need to calculate each investor's share per repayment — but investors get returns progressively. Option B is simpler but investors wait longer.

The industry terms worth knowing:

    Disbursement — platform → borrower
    Repayment — borrower → platform
    Distribution or Investor Payout — platform → investor

Your TransactionType enum likely needs all three as distinct values, because each has different accounting implications, different triggers, and different parties involved. What an activity log actually is

At its core, an activity log is an append-only record of things that happened in the system. The key word is append-only — entries are never updated or deleted, only added. This is what distinguishes it from your regular entities which mutate over time.

The contrast worth analyzing:

LoanRequest     → mutable, current state only, history is lost
ActivityLog     → immutable, every state change preserved

Your LoanRequest right now can tell you its current status. It cannot tell you when it moved from PENDING_APPROVAL to APPROVED, or who approved it, or what it looked like before. An activity log fills that gap.
The two distinct problems activity logs solve

Audit trail — who did what and when, for compliance and accountability. In a fintech system this is not optional — regulators require it. Who approved this loan? Who disbursed these funds? Who granted this user admin privileges?

Debugging and support — when something goes wrong, what sequence of events led there? This is what your operations team uses when a user says "my loan status changed unexpectedly."

These two needs pull the design in slightly different directions, which is the first evaluation decision you need to make.
The three industry approaches, evaluated

Approach 1 — Generic event log

One table, every action in the system recorded as a row:
java

@Entity
public class ActivityLog {
    private Long id;
    private String actorEmail;      // who did it
    private String action;          // what they did — "LOAN_APPROVED"
    private String entityType;      // what it affected — "LoanRequest"
    private Long entityId;          // which specific one
    private String details;         // JSON blob of relevant context
    private LocalDateTime timestamp;
}

Tradeoff: simple to implement, one table, flexible. But querying is awkward — finding all actions on a specific loan requires filtering by entityType and entityId. The details JSON blob is untyped, so you lose compile-time safety on what gets recorded.

Approach 2 — Domain-specific event tables

Separate tables per domain: LoanActivityLog, UserActivityLog, TransactionActivityLog.

Tradeoff: typed, queryable, honest about structure. But you multiply tables and risk inconsistency across them. This is what large teams with separate domain teams do.

Approach 3 — Event sourcing

The entire system state is derived from an immutable event stream. You never store current state directly — you replay events to compute it.

Tradeoff: extremely powerful for audit and debugging, but architecturally invasive. Not appropriate at your stage — it requires designing the system around events from the start.
The evaluation for your system

Approach 1 is the right fit right now. Here is why:

Your system is a single application with a small number of domain objects. The flexibility cost of a JSON details blob is low because you control both the writer and the reader. The query awkwardness is manageable because audit queries are infrequent. Approach 2 multiplies tables without enough query benefit at your scale. Approach 3 is architecturally premature.

The connection between activity logs and notifications

Right now your system records things that happen. Notifications are the next natural layer — reacting to things that happen. The two are more connected than they appear.

When you log LOAN_APPROVED, you probably also want to notify the borrower. When you log INVESTMENT_MADE, you probably want to notify the loan request owner that their funding progressed. When you log LOAN_DISBURSED, you want to notify the borrower that funds are on the way.

The evaluation question is: where does that reaction live?
The naive approach and why it breaks

The first instinct is to put it directly in the service method:
java

public LoanRequestDTO approveLoanRequest(Long requestId) {
    // ... approval logic ...
    activityLogService.log(...);
    emailService.sendApprovalEmail(borrower);  // ← added here
    pushNotificationService.notify(borrower);  // ← and here
    return LoanRequestUtils.mapLoanRequestEntityToOutput(updatedRequest);
}

This works initially but creates a serious design problem. approveLoanRequest now has four responsibilities: approve the loan, persist it, log it, and notify the borrower. Every new notification requirement means modifying business logic. If the email service is down, the approval fails. The method becomes fragile and hard to test.
The industry solution — events

This is where Spring Application Events become genuinely valuable, not just architectural complexity for its own sake.

The pattern:
java

// service publishes what happened
public LoanRequestDTO approveLoanRequest(Long requestId) {
    // ... approval logic ...
    applicationEventPublisher.publishEvent(
        new LoanApprovedEvent(updatedLoan)
    );
    return LoanRequestUtils.mapLoanRequestEntityToOutput(updatedLoan);
}

// separate listener reacts to it
@Component
public class LoanEventListener {

    @EventListener
    public void onLoanApproved(LoanApprovedEvent event) {
        activityLogService.log(LOAN_APPROVED, event.getLoan());
        notificationService.notifyBorrower(event.getLoan());
    }
}

The service now has one job — execute the business operation and announce what happened. Everything that reacts to that announcement lives elsewhere. Adding a new reaction — SMS, push notification, webhook — means adding code to the listener, not touching the business logic.
The three notification channels worth knowing

In-app notifications — a Notification table in your database. The frontend polls or uses WebSockets to show a bell icon with unread count. This is what you'd build first since you already have the database infrastructure.

Email notifications — triggered by events, sent via SMTP or a service like SendGrid. Stateless — you fire and forget.

Push notifications — mobile only, requires Firebase Cloud Messaging or similar. Not relevant until you have a mobile app.
The evaluation for your system right now

You have two features left before loan payments — activity log and disputes. The honest evaluation of sequencing is:

Building the activity log with direct service method calls is the right short-term move. It gets logging working with minimal new concepts.

But if you design the activity log using Spring events from the start, you get notifications almost for free later — the listener just grows a new line. If you hardcode logging into service methods, adding notifications later means going back and modifying every service method you already wrote.

The decision cost is low now and high later. That's the signal that the event approach is worth the upfront investment even at your scale.

The design decisions worth making before coding

Decision 1 — What goes in details?

The details field is a JSON string carrying context specific to each event type. The evaluation question is how much to store.

Minimal approach — just the IDs:
json

{"loanRequestId": 5, "approvedById": 2}

Snapshot approach — the full state at the time of the event:
json

{
  "loanRequestId": 5,
  "requestedAmount": 1000000,
  "borrowerEmail": "jay@example.com",
  "approvedById": 2,
  "approvedByEmail": "admin@example.com",
  "previousStatus": "PENDING_APPROVAL",
  "newStatus": "APPROVED"
}

The snapshot approach is significantly more valuable for audit purposes. If a loan request is later deleted or modified, the log still tells you exactly what the state was at approval time. The minimal approach requires joining back to live tables to reconstruct context — which defeats the purpose of an immutable log.

Evaluation: snapshot approach is correct for fintech. The storage cost is negligible. The audit value is high.

Decision 2 — How do you serialize the details to JSON?

You need Jackson's ObjectMapper to convert a map or object to a JSON string before storing. Spring Boot already has this configured — you just inject it.

Decision 3 — Synchronous or asynchronous listeners?

By default Spring events are synchronous — the listener runs in the same thread as the publisher, inside the same transaction. This means if the listener fails, it can roll back the business operation.

Adding @Async makes the listener run in a separate thread — the business operation completes regardless of what the listener does.

The evaluation:

Synchronous → logging failure can roll back a loan approval. Safer for audit compliance but fragile if logging has a bug.

Asynchronous → logging failure is silent, business operation always completes. Better user experience but you can lose audit entries.

For a fintech system the honest answer is synchronous for now — a logging failure should be visible, not swallowed. You can move to async later with proper error handling around the listener.

Decision 4 — Who is the actor?

Some events have a clear human actor — LOAN_APPROVED was triggered by an admin. Others are system-triggered — LOAN_FULLY_FUNDED happens automatically when an investment tips the balance. Your log needs to handle both:

private String actorEmail;   // null or "SYSTEM" for automated events
private String actorRole;    // "ADMIN", "USER", "SYSTEM"
